# 2. Transactions — Basic

## Chapter Digest

The essence of money is not the physical object but the ledger. The people of Yap Island in Micronesia used massive limestone discs as currency, yet when exchanging goods they never moved the stones. Only ownership changed hands. Even a stone that had sunk to the bottom of the ocean during a storm was still counted as belonging to its owner. The community's consensus about who owned what was the currency itself, not the object. The same is true today: when you transfer money from your bank account, no physical cash moves anywhere. Only numbers in a ledger change. Bitcoin works exactly the same way. The Bitcoin system contains no balance data for any given address. It holds only a history of which public key sent bitcoin to which public key.

Satoshi Nakamoto's whitepaper defines an electronic coin as a chain of digital signatures. Each owner transfers the coin by signing a hash of the previous transaction and the next owner's public key, then appending that signature to the end of the coin. Anyone receiving the coin can verify the chain of ownership by verifying the signatures. When Alice sends 300,000 satoshis to Bob, her signature is attached to that transaction. When Bob sends it on to Carol, his signature is attached. This unbroken sequence of signatures is the chain of digital signatures. One BTC equals 100 million satoshis (SATS), the smallest unit of bitcoin.

A Bitcoin transaction consists of inputs (vin) and outputs (vout). Inputs reference an Unspent Transaction Output (UTXO) from a previous transaction, along with a signature proving the right to spend it. Outputs specify the recipient's public key and the amount being sent. The balance shown in a wallet app is not stored anywhere in the network directly — it is the total of all UTXOs associated with a given address, calculated by a full node searching the entire transaction history.

Private keys and public keys are both numbers. A private key can generate a public key, but it is computationally infeasible to reverse the process and derive the private key from the public key. A private key is like a stamp; a public key is like the verification template made from that stamp's shape. When a signature is produced with the private key and checked against the template, it either matches or it does not. The public key also serves as the account address. In current practice, most Bitcoin transactions use a hashed form of the public key — called an address — as the destination rather than the raw public key itself.

The central problem Bitcoin had to solve is double-spending. Imagine a fraudster named Mallory who holds a single UTXO worth 30,000 satoshis and simultaneously sends it to Bob, Carol, and Dave. Prior electronic cash systems solved this by routing all transactions through a central server, which recorded them in the order received and accepted only the first, rejecting any later attempts to spend the same coin. Banks work the same way. Every transaction passes through the bank, which verifies that no double-spend is occurring before crediting the recipient. But this creates a single point of failure: if the central server is destroyed or shut down, the entire system collapses.

Bitcoin introduces no trusted third party. Instead, it makes all transactions public so that every network participant can take part in verification. No single entity monopolizes the right to validate transactions. This transparency is one of Bitcoin's core properties. In a decentralized network, however, there is no absolute time. Each computer's clock differs slightly from every other, and relying on a centralized time server would recreate the single point of failure. Even if timestamps were attached to transactions, no timestamp could be trusted in a distributed system because there is no universally agreed reference clock, and different nodes receive the same transaction at different times.

What was needed, therefore, was a method other than wall-clock time for establishing the order of transactions. The solution is described in Chapter 3 of the whitepaper: the timestamp server. Bitcoin's key innovation is building a decentralized system in which everyone can agree on the sequence of events without trusting any central authority.

Anyone who keeps the full history of every bitcoin transaction ever made is called a full node. Anyone can run a full node. Full nodes verify incoming transactions and maintain the set of unspent outputs. Wallet applications use the data provided by full nodes to find all UTXOs associated with a given address and sum them into a displayed balance. Because bitcoin has no physical form, it is harder to seize than gold and guarantees complete property rights without regard for borders.
