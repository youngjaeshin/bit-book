# Chapter 9 Technical Understanding — Elliptic Curve Cryptography and Bitcoin Payment Methods

## Elliptic Curve Arithmetic

The relationship between Bitcoin's private key and public key rests on Elliptic Curve Cryptography (ECC). Bitcoin uses a specific curve called secp256k1, defined by y² = x³ + 7.

Elliptic curve addition is defined as follows. Given two points P and Q on the curve, draw the line through them. The line intersects the curve at a third point −R; reflecting −R across the x-axis gives R = P + Q. This addition obeys commutativity and associativity. For P(x₁, y₁) and Q(x₂, y₂), compute slope m = (y₂ − y₁)/(x₂ − x₁) and derive R's coordinates algebraically. When P = Q, use the tangent line at P instead.

Scalar multiplication kG repeats elliptic curve addition k times. Using the double-and-add algorithm, computing kG for a k with n bits requires at most 2n operations — linear in the bit length of k.

## Finite Fields and the Discrete Logarithm Problem

In cryptography, points on the elliptic curve are restricted to a finite field defined by a prime p. The finite field Fp contains only integers from 0 to p − 1. Addition and multiplication are performed normally and then reduced mod p. The multiplicative inverse uses Fermat's little theorem: a⁻¹ = a^(p−2) mod p.

For secp256k1, the prime p is 2²⁵⁶ − 2³² − 977, an astronomically large number. The set of points satisfying y² ≡ x³ + 7 (mod p) forms the basis for cryptographic operations.

The discrete logarithm problem is the security foundation of ECC. Computing Q = kG from k and G takes linear time O(n). Recovering k from G and Q has no known efficient algorithm — it takes exponential time O(2ⁿ) via brute force. With k at 256 bits, this reversal is computationally infeasible.

## Deriving the Public Key

A Bitcoin private key is a 256-bit random number. The public key is Q = kG, computed via elliptic curve scalar multiplication with the generator point G whose coordinates are fixed in the secp256k1 specification. Q is a point (x, y) on the curve.

The WIF (Wallet Import Format) encoding of a private key prepends 0x80, double-SHA256 hashes the result, appends the first 4 bytes of the hash as a checksum, and Base58-encodes the whole thing. Base58 excludes the characters 0, I, O, and l to reduce the risk of misreading. A WIF private key starts with '5'.

Public keys come in two forms. The uncompressed form is 0x04 ‖ x ‖ y (520 bits). The compressed form observes that for any x, the two valid y values are one even and one odd; prefix 0x02 if y is even, 0x03 if y is odd, followed only by x (264 bits — roughly half the size). The compressed form reduces block space consumption and transaction fees. A private key intended to derive a compressed public key has 0x01 appended before the checksum is computed, producing a compressed WIF key starting with 'K' or 'L'.

## ECDSA — Elliptic Curve Digital Signature Algorithm

A trapdoor function is easy to compute in one direction but infeasible to reverse. Given k and G, computing Q = kG is straightforward; recovering k from G and Q is the discrete logarithm problem. The private key acts as the trapdoor. Because different keys are used for signing and verification, this constitutes asymmetric-key cryptography. The foundational concepts of trapdoor functions and asymmetric keys were introduced by Diffie, Hellman, and Merkle, whose Diffie-Hellman key exchange was the first publicly disclosed public-key cryptosystem.

To generate an ECDSA signature, three inputs are required: the private key k, the hash z of the unsigned raw transaction, and a freshly generated 256-bit nonce r. Compute R = rG and let Rx be R's x-coordinate. The signature s is s = r⁻¹(z + Rx · k) mod n, where n is the order of the elliptic curve group. The pair (Rx, s) constitutes the signature.

Verification uses only public information: signature (Rx, s), transaction hash z, generator G, and public key Q. Compute (z/s)G + (Rx/s)Q; if its x-coordinate equals Rx, the signature is valid. No private key exposure is needed — this is the zero-knowledge-like property of the scheme.

Reusing the nonce r, or using a nonce with insufficient randomness, leaks the private key: two signatures with the same r yield a system of two equations in two unknowns (r and k), solvable for k. The 2013 Android wallet hack exploited exactly this vulnerability.

## Schnorr Signatures

Claus-Peter Schnorr developed a more efficient signature scheme in 1989. Rather than using the modular inverse in ECDSA's signing step, Schnorr hashes the concatenation of Rx, the public key Q, and the raw transaction m together: s = r + H(Rx ‖ Q ‖ m) · k mod n. The patent expired in 2008, and Schnorr signatures were introduced to Bitcoin via the Taproot upgrade in November 2021 (block 709,632). Both ECDSA and Schnorr are currently in use.

Schnorr's most significant advantage is its native support for multisignature aggregation. Add M public keys with elliptic curve addition to obtain a single aggregate public key Q. Each of M signers contributes a partial signature si; summing them yields a single aggregate signature s. The result is indistinguishable from a single-signer transaction — no individual public keys appear in the locking script, preserving participant privacy and reducing script size and fees.

Pure M-of-M Schnorr multisig works straightforwardly, but M-of-N threshold signing (any M of N signers suffice) requires Shamir's Secret Sharing. The idea: construct a polynomial of degree M − 1 with the private key k as the constant term; sample N distinct points on the polynomial and distribute one to each participant. Recovering k requires at least M points (via Lagrange interpolation); M − 1 points reveal nothing about k.

Combining Schnorr and Shamir gives the FROST protocol (Flexible Round-Optimized Schnorr Threshold Signatures). FROST aggregates public key shares and signature shares using Lagrange weights, enabling any M of N participants to produce a signature verifiable by the aggregate public key.

## Bitcoin Payment Methods

**Legacy Address and P2PKH:** Hash the public key with SHA256 then RIPEMD160, prepend the version byte 0x00, append a 4-byte checksum, and Base58-encode to get a legacy address beginning with '1'. P2PKH (Pay-to-Public-Key-Hash) stores the address in the locking script and requires the spender to present both a signature and the full public key in the unlocking script. The public key is revealed only at spend time, making it safer than P2PK.

**P2MS and P2SH:** P2MS (Pay-to-Multi-Sig) embeds all public keys directly in the locking script, which burdens full nodes' UTXO set storage. P2SH (Pay-to-Script-Hash) addresses this by hashing the original locking script (the redeem script) and storing only that hash in the locking script. The redeem script itself is presented in the unlocking script when spending. P2SH addresses use the prefix 0x05 and always begin with '3'.

When the script engine encounters OP_HASH160 and OP_EQUAL in combination, it recognizes a P2SH transaction and treats the presented redeem script as a new locking script to evaluate. This lets the sender use a simple hash-based address while the complex multi-signature logic is managed entirely by the recipient.

Beyond these, segwit introduced P2WPKH and P2WSH, and Taproot introduced P2TR addresses (beginning with 'bc1p'). Each advancement improves efficiency, security, and privacy simultaneously — a trajectory that continues with ongoing protocol development.
