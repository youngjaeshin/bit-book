# 7. Reclaiming Disk Space — Basic

Chapter 7 of the Bitcoin whitepaper introduces the Merkle tree structure as the solution to the blockchain's storage problem and argues that decentralization is the foundational value that drives Bitcoin's design choices.

## The Merkle Tree: Deleting Transaction Data Without Changing Block Hashes

As of 2024, running a full node requires over 600 GB of storage. Once a transaction output has been spent and many blocks have been stacked on top of it, is it necessary to keep that transaction's data forever? The problem is that if a block was hashed including its full transaction data, removing any transaction would change the block's hash entirely — a consequence of the avalanche effect in hash functions. The Merkle tree structure solves this.

A transaction ID (txid) is the hash of a transaction's data. Because a txid can only be produced by hashing that exact transaction, storing the txid is equivalent to proving the transaction existed. Therefore, when computing a block's hash, it is not necessary to include the raw transaction data — only the txids need to be included.

The Merkle tree takes this further. Transaction txids are paired and hashed together. The resulting hashes are then paired and hashed again. This process repeats until a single 32-byte hash remains: the Merkle root. If the number of hashes is odd at any step, the last hash is duplicated to make it even. The structure resembles an inverted tree — hence "Merkle tree," also called a binary hash tree. The Merkle root is placed in the block header, and only the block header is hashed when computing the block's hash.

With 4,000 transactions in a block averaging 250 bytes each, the full transaction data is 1 MB. Storing only txids reduces this to 128 KB (4,000 × 32 bytes). Computing and storing only the Merkle root reduces it to 32 bytes — roughly 1/30,000th of the original data. Deleting the underlying transaction data and retaining only the Merkle root in the block header leaves the block's hash unchanged. Nodes that prune transaction data this way are called SPV nodes or light nodes.

## Why Merkle Trees Specifically?

Hashing all transactions together into a single digest would also allow block hashes to remain stable after transaction data is deleted — so why go through the paired, iterative structure of a Merkle tree?

Two reasons. First, Merkle trees enable efficient transaction inclusion proofs for light nodes. If all transactions were hashed together into one digest, a light node wanting to verify that a specific transaction is in a block would have to download all block transaction data to recompute the hash. With a Merkle tree, the light node only needs the hashes along the path from the transaction to the Merkle root — called a Merkle proof — a logarithmically small amount of data. This is the basis of Simplified Payment Verification (SPV), covered in the next chapter.

Second, Merkle trees excel at tamper detection. If any transaction in a block is altered, only the branch of the tree containing that transaction changes. By comparing branches, the tampered transaction can be pinpointed quickly without examining every transaction.

## Storage, Decentralization, and Philosophy

The block header is approximately 80 bytes. At one block every ten minutes, one year of block headers requires about 80 × 6 × 24 × 365 ≈ 4.2 MB — less than a single low-resolution smartphone photo. Storing full transaction data for all blocks requires about 600 GB as of 2024, addressable with an external SSD costing around 100,000 Korean won (~$75 USD). So why did Satoshi Nakamoto go to such lengths to minimize storage requirements?

The answer is decentralization. Every additional node that validates transactions strengthens the network. Any cost barrier to running a node — even 100,000 won — excludes students, people in difficult circumstances, and those in underdeveloped regions with limited access to computers. Bitcoin is not a monetary system designed for any privileged group; it must be accessible to everyone who wants to participate in transaction validation.

Bitcoin is a system that bets everything on decentralization. A currency or payment system with better performance but without decentralization is not a system for everyone. Bitcoin's on-chain layer is slow by design — it is a final settlement layer, comparable to the interbank clearing houses that take days to settle. Day-to-day transactions are handled by the Lightning Network at the second layer, achieving near-instant settlement for negligible fees.

A faction within the Bitcoin community once argued that decentralization should be partially sacrificed for the sake of exchange-medium functionality — that some people being excluded was an acceptable tradeoff for serving more people faster. This produced one of the fiercest conflicts in Bitcoin's history, known as the Blocksize War. The outcome — and its meaning — is examined in the deep-dive section.
