# 3. Timestamp Server — Technical

## Chapter Digest

The first real-world commercial transaction conducted in bitcoin took place on May 22, 2010, when Florida programmer Laszlo Hanyecz posted on the Bitcoin Talk forum offering 10,000 BTC for two large pizzas. Jeremy Sturdivant, living in the UK, accepted the offer, ordered two Papa John's pizzas to Hanyecz's address using his own credit card, and received the bitcoin. The transaction is recorded in the block at height 57,043, and the Bitcoin community commemorates May 22 every year as Bitcoin Pizza Day. This block's raw data provides an ideal case study for reading actual Bitcoin block structure.

A block is divided into a block header and a block body. The block body contains transaction data. The block header contains exactly six metadata fields and is always exactly 80 bytes in size. The six fields are: version (4 bytes), previous block hash (32 bytes), Merkle root (32 bytes), timestamp (4 bytes), difficulty target bits (4 bytes), and nonce (4 bytes). The block hash is produced by applying the SHA256 hash function twice — double-hashing — to these 80 bytes. Because the block header incorporates the Merkle root (a cryptographic summary of all transactions in the block) and the hash of the previous block, the resulting block hash is effectively a compressed fingerprint of the block's entire content and its entire ancestry.

To read raw Bitcoin data, one must understand number bases and byte ordering. Decimal notation uses digits 0–9 in each position; binary uses only 0 and 1; hexadecimal (hex) uses 0–9 plus A–F, where A equals 10 and F equals 15. One hex digit represents four binary bits, so one byte (8 bits) is written as exactly two hex characters. The number 21 in hex is 0x15; the number 210 is 0xD2.

Serialization is the process of converting data into a byte-by-byte stream for network transmission or storage. Bitcoin data uses both little-endian and big-endian byte ordering. In big-endian ordering — the format humans naturally read — the most significant byte comes first. In little-endian ordering, the least significant byte comes first, which is convenient for computers because they can store incoming bytes sequentially without knowing the total data size in advance. A three-byte value like 0x280DE8 stored in big-endian reads 28 0D E8; in little-endian it reads E8 0D 28. Because Bitcoin mixes both conventions, one must know which encoding applies to each field before interpreting the data.

The serialized block header for block 57,043 is: 010000008095126f08f377b143410cc3c5da3f3d25732d014dc9da1855e8e71300000000 5a1b723ae6c479056af838a6ea40db6f4acee39f262bf49864cd98f511221d5c7f1ff84b 249c151c23af360b. Splitting this 160-character (80-byte) string into its six fields yields version, previous block hash, Merkle root, timestamp, difficulty bits, and nonce — all stored in little-endian. Reading the version field 01000000 by reversing byte pairs gives 00000001, meaning version 1. Reversing the previous block hash field produces 0000000013e7e85518dac94d012d73253d3fdac5c30c4143b177f3086f129580, which matches the hash of block 57,042 visible in any block explorer.

The timestamp field 7f1ff84b reversed gives 0x4bf81f7f. Converting this Unix timestamp — seconds elapsed since January 1, 1970 00:00:00 UTC — to a calendar date yields May 22, 2010, confirming the pizza transaction date in the raw data itself. As noted earlier, this timestamp is not used to establish the causal order of transactions; that order comes from the block chain structure. The timestamp serves primarily as an input to the difficulty adjustment algorithm. Because node clocks differ across a distributed network, Bitcoin's protocol allows block timestamps to deviate from real time by up to two hours.

The difficulty bits field 249c151c reversed gives 0x1c159c24. This compact encoding represents the mining target: 0x1c specifies the number of bytes in the full target value, and 0x159c24 is the significand. A miner must produce a block header whose double-SHA256 hash, when read as a number, is less than this target. The nonce field 23af360b reversed gives 0x0b36af23, which is 188,133,155 in decimal. Miners increment the nonce from zero, hashing the header each time, until they find a hash below the target. This is proof-of-work, covered in depth in the next chapter.

Block hashes output from SHA256 are themselves in little-endian, so the raw output bytes must be reversed to match the representation shown in block explorers. Double-hashing the 80-byte serialized header of block 57,043 and reversing the result produces 00000000152340ca42227603908689183edc47355204e7aca59383b0aaac1fd8. The leading zeros in this hash are not coincidental: the difficulty target requires that a valid block hash begin with a certain number of zero bytes. The more leading zeros required, the more hashing attempts — and therefore the more energy — were needed to find a valid block. The number of leading zeros is a direct measure of the computational work invested.

Bitcoin uses double-SHA256 rather than single-SHA256 to defend against length-extension attacks. A single application of SHA256 is vulnerable to a class of attack in which an adversary can produce a valid hash for a message with arbitrary data appended, without knowing the original message. Hashing the SHA256 output a second time eliminates this vulnerability.

Every field in every block is publicly accessible and independently verifiable. Anyone can query a full node — or a public interface such as chainquery.com/bitcoin-cli — to retrieve raw block data, parse the header fields, perform the double-SHA256 computation, and confirm that the result matches the canonical block hash. This is what "don't trust, verify" means in practice: not taking any party's word for the state of the ledger, but deriving the truth directly from the mathematics. The timestamp server's technical implementation — 80-byte headers chained by hash, double-hashed for their identifier, with all data openly serialized and broadcast — is the engineering foundation that makes a trustless, tamper-resistant, globally shared record of time-ordered events possible without any central authority.
