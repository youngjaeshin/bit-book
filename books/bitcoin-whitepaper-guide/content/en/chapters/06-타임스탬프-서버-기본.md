# 3. Timestamp Server — Basic

## Chapter Digest

The first key to understanding how Bitcoin works is the hash function. Hash functions share four essential properties. The first is determinism: the same input always produces the same output. The second is fixed-length output: regardless of whether the input is two characters or a building's worth of data, the output is always the same length. Bitcoin's primary hash function, SHA256, always produces a 256-bit result expressed as 64 hexadecimal characters, no matter the size of the input. The third property is the avalanche effect: changing even a single character of the input produces a completely different output. The fourth is one-wayness: it is computationally infeasible to reverse-engineer the input from the output. This one-way property is why hash functions are so widely used in security — storing a hashed password rather than the password itself being a familiar example. A hash output serves as a compressed fingerprint of its input data, uniquely representing that data in a fixed-size form.

A block is a container for verified transaction data. Before a transaction can enter a block, it must pass validation: the validator checks that the referenced previous transaction actually exists, then uses the public key in that earlier transaction's output to confirm that the new transaction's signature is valid. Only transactions that pass this check are admitted to the block. Each block also carries a block header, which records metadata about the block: the Merkle root (a hash summarizing all transactions in the block), a rough timestamp of creation, difficulty-related values (the nonce and bits fields), and crucially, the hash of the previous block.

The identifier for a block is the block hash, produced by hashing the block header. Because the block header contains the Merkle root — a compressed digest of all the block's transactions — the block hash effectively encapsulates everything in the block. The critical element is the inclusion of the previous block's hash in the header. When the second block's header is hashed with the first block's hash embedded in it, the resulting hash implicitly contains all of the first block's data. The third block includes the second block's hash, which already contains the first block's data, so the third block's hash implicitly contains all prior data. As this chain extends, the blocks become inseparably linked. This is the blockchain.

This structure means that altering even one byte in any past block changes that block's hash, which changes every subsequent block's hash in a cascade. The more blocks have been added on top, the more computationally overwhelming it becomes to alter earlier data. Satoshi Nakamoto himself described Bitcoin as "a distributed secure timestamp server for transactions." The block hash is the timestamp. Transactions included in the same block are treated as simultaneous; transactions in the next block are treated as later. Block order becomes the definition of time, and the ordering of transactions is thereby determined without any centralized clock. This is the mechanism that resolves the double-spending problem in a decentralized system: the timestamp server creates an ordering of events that everyone in the network can agree on without trusting any single authority.
