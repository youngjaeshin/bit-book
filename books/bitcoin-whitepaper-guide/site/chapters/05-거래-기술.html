2. Transactions — Technical

Chapter Digest

The first bitcoin transfer in history took place on January 12, 2009, when Satoshi Nakamoto sent 10 BTC to Hal Finney. That transaction is recorded in the block at height 170 and provides a concrete window into how the UTXO model works in practice.

A raw transaction contains several fields: txid (a unique transaction identifier derived by hashing the transaction data), size (the byte length of the data), version (the ruleset version used to process it), locktime (a value that, when set above zero, prevents the transaction from being included until a specified time), vin (an array of inputs), and vout (an array of outputs). Each entry in vin references a previous transaction by its txid, specifies which output of that transaction to consume via a vout index, and includes a scriptSig — the unlocking signature. Each entry in vout specifies a value in BTC and a scriptPubKey — the locking script that defines the conditions for spending. In the first transfer, the input references a coinbase transaction, which is a mining reward with no prior UTXO input. The two outputs send 10 BTC to Hal Finney's public key and return 40 BTC to Satoshi's public key as change.

Bitcoin uses a stack-based scripting language to define and verify spending conditions. A script is read left to right: when a value is encountered it is pushed onto the stack; when an operator is encountered it pops values from the top of the stack, performs its operation, and pushes the result back. For example, the script "3 5 OP_ADD 8 OP_EQUAL" pushes 3 and 5, then OP_ADD replaces them with 8, then 8 is pushed again, and OP_EQUAL checks whether the two top values match, returning TRUE. This language is deliberately Turing-incomplete: it has no loops, no arbitrary jumps, and stores nothing between executions. Simplicity is a security feature — a Turing-complete language exposes far more attack surface. The contrast with Ethereum's smart contract language is instructive: in 2016, a reentrancy vulnerability in the DAO contract — enabled precisely by loop capability — allowed an attacker to drain approximately 3.6 million ETH.

To spend a UTXO, a node concatenates the scriptSig from the new transaction's input with the scriptPubKey from the output being consumed, then executes the combined script. In the P2PK (Pay-to-Public-Key) format used in the earliest transactions, this produces the combined script: `<signature> <public key> OP_CHECKSIG`. The OP_CHECKSIG operator pops the public key and the signature from the stack, verifies that the signature was produced by the private key corresponding to that public key, and returns TRUE if valid or FALSE otherwise. A valid signature proves that the spender holds the private key — without the private key, no valid signature can be produced, and the locking script cannot be satisfied. The scriptPubKey is the lock; the scriptSig is the key. A signature is more like a handwriting sample than a stamp: if the transaction content changes, the old signature no longer validates and a new one must be produced.

2. 거래 — 기술

Chapter Digest

비트코인 역사상 최초의 송금 거래는 2009년 1월 12일 사토시 나카모토가 할 피니에게 10 BTC를 보낸 것으로, 블록 높이 170에 기록되어 있다. 이 거래 데이터를 통해 UTXO 모델이 실제로 어떻게 구현되는지 확인할 수 있다.

실제 거래 데이터는 txid(거래 번호), size(데이터 크기), version(규칙 버전), locktime(잠금 시간), vin(입력값 배열), vout(출력값 배열)으로 구성된다. txid는 거래 데이터를 해싱하여 얻은 고유 번호다. vin 안에는 소비할 이전 거래의 txid, 해당 거래의 몇 번째 출력값을 쓸지 나타내는 vout 인덱스, 그리고 서명인 scriptSig가 들어간다. vout에는 전송 금액인 value와 잠금 스크립트인 scriptPubKey가 들어간다. 최초의 거래 데이터에서 입력값은 이전에 채굴 보상으로 사토시가 받은 코인베이스 거래를 참조하고, 출력값에는 할 피니에게 보내는 10 BTC와 사토시에게 거슬러주는 40 BTC가 각각 들어있다.

비트코인은 스크립트 언어라는 스택 기반의 프로그래밍 언어를 사용한다. 스크립트는 왼쪽에서 오른쪽으로 순서대로 실행되며, 값이 나오면 스택에 쌓고 연산자가 나오면 스택 상단의 값들을 꺼내 연산한다. 예를 들어 '3 5 OP_ADD 8 OP_EQUAL'을 실행하면 3과 5를 스택에 쌓고, OP_ADD가 둘을 더해 8로 만들고, 다시 8을 스택에 쌓은 뒤 OP_EQUAL이 두 값이 같은지 확인해 TRUE를 반환한다. 이 언어는 튜링 불완전하다. 루프나 임의 점프가 없고, 실행 중 데이터가 저장되지 않는다. 단순하기 때문에 공격 취약점이 훨씬 적다. 이더리움의 복잡한 스마트 콘트랙트 언어가 2016년 DAO 해킹 사건에서 무한 루프 취약점으로 364만 이더리움이 탈취된 것과 대비된다.

잠금 스크립트(scriptPubKey)는 UTXO를 소비하기 위한 조건을 담은 코드이며, 해제 스크립트(scriptSig)는 그 조건을 만족하는 증거를 담은 코드다. 서명을 검증할 때는 해제 스크립트에 이어 잠금 스크립트를 연결하여 실행한다. 최초의 거래 형태인 P2PK(Pay-to-Public-Key) 방식에서 이 둘을 합치면 '<서명> <공개키> OP_CHECKSIG' 형태가 된다. OP_CHECKSIG는 스택 최상단의 공개키를 이용해 그 아래의 서명을 검증하고 TRUE 또는 FALSE를 반환한다. 서명이 검증된다는 것은 해당 공개키에 대응하는 개인키를 가진 사람이 거래를 승인했다는 뜻이다. 개인키를 모르면 유효한 서명을 만들 수 없고, 잠금 스크립트를 해제할 수 없다. 잠금 스크립트가 자물쇠라면 해제 스크립트는 열쇠다. 서명은 도장이 아니라 필적 확인에 가깝다. 거래 내용이 바뀌면 새로운 서명을 제시해야 하기 때문이다.
