10. Privacy — Technical Understanding

10. Privacy — Technical Understanding

The Principles of Hierarchical Deterministic (HD) Wallets

To protect privacy, a different Bitcoin address should be used for each transaction. However, randomly generating and storing a separate private key, public key, and address pair for every transaction would be enormously cumbersome. Today this problem is solved by using mnemonic phrases and the standard derivation principles of hierarchical deterministic (HD) wallets, which generate a virtually unlimited number of addresses from a single seed.

The basic principle of HD wallets is as follows. First, an entropy value of 128 or 256 random bits is generated. This entropy is converted into a mnemonic — a sequence of memorable English words. A 128-bit entropy yields 12 words; 256-bit entropy yields 24 words. The mnemonic is then hashed 2,048 times using the HMAC-SHA512 function (a process called PBKDF2) to produce a 512-bit seed. That seed is hashed again with HMAC-SHA512 using the key string "Bitcoin seed" (capital B required), producing a 512-bit result: the first 256 bits become the master private key and the last 256 bits become the master chain code. The master private key combined with the master chain code is called the extended master private key; the master public key combined with the master chain code is called the extended master public key.

From the extended master private key, up to approximately 4.2 billion child private keys can be derived. Each child private key can in turn generate another ~4.2 billion grandchild keys, and so on indefinitely. Because this hierarchy is deterministic, a single mnemonic can produce an infinite number of private keys — hence the name hierarchical deterministic wallet. Without the chain code, an outsider cannot discover the relationship between addresses, so all addresses derived from one mnemonic appear completely independent to any observer.

A particularly powerful feature of HD wallets is that child public keys can be generated without ever exposing the private key. Given only the extended master public key (master public key + master chain code), one can derive all child public keys. This is possible because the elliptic curve scalar multiplication satisfies the distributive law. As a result, an air-gapped signing device can export only the extended public key to a watch-only wallet, which can then track balances across all derived addresses without having any ability to spend funds.

Generating Entropy and Mnemonic Phrases

Following the BIP-0039 standard, a checksum is appended to the entropy before conversion to a mnemonic. The checksum length in bits equals the entropy length divided by 32. For 256-bit entropy, an 8-bit checksum is appended, giving 264 bits total. This is split into 24 groups of 11 bits each, and each group maps to one word in a 2,048-word list. The list is carefully curated: no two words share the same first four characters, and easily confused words are excluded.

When deriving a seed from the mnemonic, the HMAC-SHA512 key is the string "mnemonic" followed optionally by a passphrase. The passphrase acts as a salt. Even a single character difference produces an entirely different seed and therefore an entirely different set of addresses. This makes passphrases a powerful additional security layer, but they must be recorded carefully because any error irrecoverably changes the wallet.

Address Derivation

There are two methods for deriving child keys from a parent key. In the **hardened child key derivation** method, the parent private key (with a 0x00 prefix) and a 4-byte index are used as the HMAC-SHA512 message, while the parent chain code serves as the key. Because this method uses the private key, child public keys cannot be derived from the extended public key alone. In the **normal child key derivation** method, the parent public key replaces the parent private key in the message. Since the elliptic curve distributive law holds, child public keys can be derived directly from the extended public key, enabling watch-only functionality.

Indexes occupy 4 bytes (32 bits). If the leading bit is 0, the key is a normal child; if it is 1, the key is a hardened child. Hardened child indexes start at 0x80000000 and are denoted with an apostrophe (e.g., m/0' for the first hardened child of the master key). Each derivation method supports ~2.1 billion keys, giving ~4.2 billion total children per parent.

Bitcoin Address Formats

Public keys derived via HD wallet paths can be expressed in several address formats. Following **BIP-0044**, a public key is double-hashed with SHA256 then RIPEMD160, prefixed with 0x00, and Base58-encoded to produce a Legacy (P2PKH) address starting with "1".

SegWit addresses use **Bech32** encoding, which uses only digits and lowercase English letters, excluding easily confused characters such as "1", "i", "o", and "u". The prefix "bc" identifies the Bitcoin mainnet. Under **BIP-0084**, a public key hash preceded by SegWit version 0x00 produces a native SegWit (P2WPKH) address starting with "bc1q". Under **BIP-0086**, the Taproot address uses SegWit version 0x01, producing addresses starting with "bc1p".

For backward compatibility with pre-SegWit nodes, the SegWit locking script can be wrapped in a P2SH script. This **nested SegWit** approach (BIP-0049) produces an address starting with "3", identical in appearance to ordinary P2SH addresses.

Derivation Path Notation

A derivation path like `m/84'/0'/0'/0/10` encodes the full history of key derivation. Reading left to right: `m` is the master private key; `84'` is the 84th hardened child (purpose: native SegWit per BIP-0084); `0'` is the 0th hardened grandchild (coin type: Bitcoin); `0'` is the 0th hardened great-grandchild (account 0); `0` is the 0th normal great-great-grandchild (external/receive chain); and `10` is the 10th address index.

Depth 1 encodes purpose (address format), depth 2 encodes coin type (Bitcoin = 0'), depth 3 encodes account number, depth 4 encodes whether the address is for receiving (0) or for change (1), and depth 5 is the address index. The final two depths use normal derivation, so the extended public key alone can generate all addresses for monitoring purposes.

Extended keys are serialized as: version (4 bytes), depth (1 byte), parent fingerprint (4 bytes), child index (4 bytes), chain code (32 bytes), key data (33 bytes), and checksum (4 bytes), then Base58-encoded. The prefixes xprv/xpub apply for BIP-0044, yprv/ypub for BIP-0049, and zprv/zpub for BIP-0084.

10. 프라이버시 — 기술적 이해

계층 결정적 지갑(HD 지갑)의 원리

프라이버시를 지키려면 거래마다 서로 다른 비트코인 주소를 사용해야 한다. 그런데 거래할 때마다 개인키·공개키·주소 쌍을 랜덤하게 만들고 전부 보관하는 일은 매우 번거롭다. 이 문제를 해결하기 위해 현재는 니모닉과 계층 결정적 지갑(HD 지갑)의 표준 파생 원리를 활용하여, 하나의 시드에서 사실상 무한한 주소를 만드는 방법을 사용한다.

HD 지갑의 간단한 원리는 다음과 같다. 먼저 128비트 혹은 256비트의 무작위 수인 엔트로피를 생성한다. 이 엔트로피를 외우기 쉬운 영어 단어인 니모닉으로 변환한다. 128비트 엔트로피는 12개 단어, 256비트 엔트로피는 24개 단어의 니모닉이 된다. 니모닉을 PBKDF2 함수, 즉 HMAC-SHA512 함수를 2,048번 반복 해싱하는 과정을 거쳐 512비트의 시드를 만든다. 이 시드를 다시 HMAC-SHA512 함수로 해싱하면 앞 256비트는 마스터 개인키, 뒤 256비트는 마스터 체인코드가 된다. 마스터 개인키와 마스터 체인코드를 합친 것을 확장 마스터 개인키라 하고, 마스터 공개키와 마스터 체인코드를 합친 것을 확장 마스터 공개키라 한다.

확장 마스터 개인키로부터 최대 약 42억 개의 자식 개인키를 생성할 수 있다. 각 자식 개인키는 다시 각각 약 42억 개의 손자 개인키를 만들 수 있으므로, 하나의 니모닉으로 계층적으로 무한한 개인키를 생성할 수 있다. 이 때문에 이 구조를 계층 결정적 지갑이라 부른다. 외부인은 체인코드 없이는 여러 주소 간의 연관성을 발견할 수 없으므로, 각 주소는 모두 서로 다른 소유자의 주소처럼 보인다.

HD 지갑이 특히 뛰어난 점은, 개인키를 공개하지 않고도 자식 공개키들을 생성할 수 있다는 것이다. 확장 마스터 공개키, 즉 마스터 공개키와 마스터 체인코드만 있으면 자식 공개키들을 만들어낼 수 있다. 이는 타원곡선 연산에서 분배 법칙이 성립하기 때문에 가능하다. 이 성질 덕분에 에어갭 지갑에서 와치온리 지갑으로 확장 공개키만 내보내면, 와치온리 지갑은 잔액을 조회할 수 있지만 비트코인을 다른 곳으로 보내지는 못한다.

엔트로피와 니모닉의 생성 과정

BIP-0039 표준에 따라 니모닉을 만들려면 먼저 엔트로피에 체크섬을 붙인다. 체크섬의 비트 수는 엔트로피 비트를 32로 나눈 값이다. 256비트 엔트로피라면 8비트 체크섬이 붙어 264비트가 되고, 이를 11비트씩 잘라 24개의 니모닉 단어에 대응시킨다. 11비트는 0부터 2,047까지 총 2,048가지 경우의 수를 가지므로 니모닉 단어 목록은 2,048개로 이루어진다. 이 목록에서 처음 4글자가 겹치는 단어는 없으며, 지나치게 유사한 단어도 배제되어 있다.

니모닉에서 시드를 추출할 때는 HMAC-SHA512 함수의 키로 "mnemonic"이라는 문자열에 패스프레이즈를 덧붙인 값을 사용한다. 패스프레이즈는 일종의 솔트 역할을 한다. 패스프레이즈가 한 글자라도 다르면 완전히 다른 시드와 지갑이 생성되므로 매우 주의해야 한다.

HD 지갑의 주소 파생 원리

시드를 HMAC-SHA512 함수로 해싱할 때는 키로 "Bitcoin seed"(대문자 B 유의)를 사용한다. 결과값 512비트의 앞 256비트는 마스터 개인키, 뒤 256비트는 마스터 체인코드가 된다.

부모 개인키에서 자식 개인키를 파생하는 방법은 두 가지다. 강화 자식키 파생 방법은 부모 체인코드를 HMAC의 키로 사용하고, 부모 개인키 앞에 0x00을 붙인 값과 인덱스를 메시지로 사용한다. 이 방법은 개인키를 사용하므로 확장 공개키만으로는 자식 공개키를 생성할 수 없다. 일반 자식키 파생 방법은 부모 개인키 대신 부모 공개키를 사용하는 것으로, 확장 공개키만으로 자식 공개키들을 생성할 수 있다. 타원곡선의 분배 법칙이 이를 가능하게 한다.

인덱스는 4바이트(32비트)로, 맨 앞 비트가 0이면 일반 자식키, 1이면 강화 자식키다. 강화 자식키 인덱스는 0x80000000부터 시작하며 아포스트로피(')로 표기한다. 마스터 개인키(m)로부터 나온 첫 번째 강화 자식 개인키는 m/0'로 표기한다. 각 파생 방법에서 일반과 강화 자식키 각각 약 21억 개씩, 총 약 42억 개의 자식 개인키를 생성할 수 있다.

여러 가지 비트코인 주소 형식

HD 지갑에서 파생된 공개키는 다양한 주소 형식으로 표현될 수 있다. BIP-0044를 따르면 공개키를 P2PKH 방식으로 SHA256, RIPEMD160 이중 해싱 후 맨 앞에 0x00을 붙이고 Base58로 인코딩한 레거시 주소가 되며, 이 주소는 1로 시작한다.

세그윗 주소는 Bech32 인코딩 방식을 사용한다. Bech32는 숫자와 영어 소문자만 사용하며, 헷갈릴 수 있는 i, 1, o, u는 제외된다. 맨 앞에는 비트코인 메인 네트워크임을 나타내는 식별자 'bc'가 붙는다. BIP-0084를 따르면 공개키 해시 앞에 세그윗 버전 0x00이 붙어 bc1q로 시작하는 네이티브 세그윗(P2WPKH) 주소가 된다. BIP-0086을 따르는 탭루트 주소는 세그윗 버전 0x01이 붙어 bc1p로 시작한다.

구버전 지갑과도 호환되도록 세그윗 잠금 스크립트를 P2SH로 감싼 네스티드 세그윗 방식도 있다. 이 방식(BIP-0049)을 따르면 맨 앞에 버전 번호 0x05를 붙이고 Base58로 인코딩하여 3으로 시작하는 주소가 만들어진다.

파생 경로 표현

HD 지갑에서 특정 주소가 어떻게 파생되었는지는 파생 경로로 표현한다. 예를 들어 m/84'/0'/0'/0/10은 마스터 개인키(m)에서 84번째 강화 자식키(목적: 네이티브 세그윗), 그 아래 0번째 강화 손자키(코인 종류: 비트코인), 그 아래 0번째 강화 증손자키(계정 0번), 그 아래 0번째 일반 고손자키(수신 주소), 10번째 현손자 주소를 의미한다.

각 깊이의 의미는 다음과 같다. 깊이 1은 목적(주소 형식)을, 깊이 2는 코인 종류를, 깊이 3은 계정 번호를, 깊이 4는 수신 주소인지 잔돈 주소인지를(0이면 수신, 1이면 잔돈), 깊이 5는 주소 인덱스를 나타낸다. 마지막 두 깊이는 일반 자식키 파생 방식을 사용하므로 확장 공개키만으로 모든 주소를 추적할 수 있다.

확장 개인키와 확장 공개키를 표현할 때는 버전 정보, 깊이, 부모 공개키 해시, 자식 인덱스, 체인코드, 키 정보, 체크섬을 이어붙인 뒤 Base58로 인코딩한다. BIP-0044에서는 xprv/xpub, BIP-0049에서는 yprv/ypub, BIP-0084에서는 zprv/zpub 접두사가 붙는다.
